🔍 How Detection Works

Edge Detection

For each edge (top / right / bottom / left), the engine scans perpendicular to the expected edge direction within a search band. It finds the strongest luma delta per scanline, then scores by support (consistency), alignment (low variance), and direction (consistent sign).

Corner Verification

At each of the 4 corners (intersection of detected edges), a small region is scanned both vertically and horizontally. The corner score = √(vertical_support × horizontal_support) — requiring evidence in both directions.

Interior/Background Contrast

Samples luma inside the detected card area vs. outside — a real card should have different luminance from the background. This prevents false positives from blank frames.

Motion Detection

Mean Absolute Difference (MAD) between consecutive frame luma buffers — low MAD = card is still. Double-buffer swap avoids allocation per frame.

Aspect Ratio Validation

Detected width/height ratio is compared to the standard ID-1 ratio (85.6 / 53.98 ≈ 1.586). The log-ratio error is clamped to [0,1] via configurable tolerance.

Hysteresis & Grace Frames

Entering a state requires stricter thresholds than leaving it (prevents flicker). Grace miss frames allow brief instability before resetting the readiness counter.

🧠 Key Design Decisions

  • No ML: all detection is pixel heuristics — works offline, zero latency, no model download
  • Double-buffer luma swap: zero-allocation after first frame — critical for smooth 15 FPS on mobile
  • Offscreen canvas: analysis runs at 381×240 (not native), saving ~15× pixel processing vs 1080p
  • Grace frames: prevents detection flicker from hand tremor without sacrificing responsiveness
  • Stale request guard: cameraRequestIdRef counter prevents async race conditions on rapid camera restart
  • All Thai UI: consistent with <html lang="th"> — designed for Thai ID card use case

⚠️ Limitations (POC Scope)

  • Detects card-shaped objects only — does not verify it's a real ID card
  • Mock validation is random 50% pass/fail — replace with actual OCR/API
  • No Web Worker — pixel analysis runs on main thread (fine at 381×240)
  • Hardcoded luminance thresholds — may need tuning for extreme lighting
  • Path2D used for guide overlay (silent no-op on very old browsers)
